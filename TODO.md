# DoubleTrouble TODO

## Идея проекта

DoubleTrouble - локально хостящийся фронтенд/веб-клиент для совместных ролевых игр в стиле SillyTavern, рассчитанный на 2+ пользователей и одного или нескольких ботов.

Пользователь запускает приложение на своем устройстве. Веб-интерфейс открывается в браузере по адресу и порту из конфигурации. Другие участники подключаются к этой же сессии по указанному адресу, если сеть и настройки доступа это позволяют.

## Базовые цели

- [ ] Локальный запуск приложения на устройстве пользователя.
- [ ] Веб-интерфейс, доступный через браузер.
- [ ] Поддержка 2+ игроков в одной ролевой сессии.
- [ ] Поддержка ботов/персонажей, участвующих в диалоге.
- [ ] Конфигурационный файл для адреса, порта и основных параметров запуска.
- [ ] Архитектура, допускающая развитие фронтенда отдельно от серверной части.
- [ ] Хранение сценариев, персонажей, истории сообщений и настроек.
- [ ] Обязательная совместимость с форматами SillyTavern: персонажи, чаты, пресеты.
- [ ] Более компактная и понятная архитектура, чем у SillyTavern, без лишнего исторического нагромождения.
- [ ] Удобный адаптивный интерфейс для компьютера, планшета и телефона.
- [ ] Базовый режим: пошаговая ролевая игра для двух людей и бота/ботов.
- [ ] Архитектурная поддержка авторизации, даже если в первом MVP вход будет простым.

## Принятые решения

- [x] Frontend-стек: React + TypeScript + Vite. Причина: удобно разрабатывать, легко сделать красивый интерфейс, можно собрать статикой и отдавать из Python-сервера.
- [x] Backend-стек: Python + FastAPI. Причина: хороший API-слой, нативная поддержка WebSocket, простая отдача frontend-сборки.
- [x] Конфигурация: YAML, чтобы быть ближе к SillyTavern и сохранить читаемость.
- [x] Хранение: file-first, совместимое с SillyTavern. SQLite пока не основной формат, но может быть добавлен позже как индекс/кеш для ускорения поиска и списков.
- [x] Боты: адаптерная архитектура по примеру SillyTavern, но сам provider/endpoint/model задает пользователь.
- [x] Совместимость с SillyTavern обязательна для персонажей, чатов и пресетов.
- [x] MVP: без режима мастера/ведущего, сначала базовая пошаговая сессия для двух игроков.
- [x] Интерфейс должен быть адаптивным с самого начала: desktop и mobile являются обязательными сценариями.
- [x] Несколько комнат/кампаний не нужны как пользовательская функция в MVP, но архитектуру надо сразу делать session-based, чтобы добавить это позже без переписывания ядра.
- [x] Код должен быть читаемым: явные слои, классы и модели вместо свалки процедурной логики.
- [x] `config/config.yaml` должен хранить только системные/админские параметры, которые обычный пользователь обычно не трогает.
- [x] Пользовательские настройки провайдера/бота должны редактироваться через frontend и храниться в `data/default-user/settings.yaml`.

## Открытые вопросы

- [x] Какой frontend-стек использовать: React + TypeScript + Vite.
- [x] Нужна ли авторизация игроков или достаточно локальной ссылки/комнаты: нужна архитектурная поддержка авторизации, но MVP может стартовать с простого входа по имени.
- [x] Боты должны подключаться к локальным моделям, внешним API или к обоим вариантам: ориентир на SillyTavern, нужна адаптерная поддержка разных backend-ов.
- [x] Нужна ли поддержка нескольких одновременных комнат/кампаний: для MVP одна активная сессия, но модель должна позволять добавить несколько позже.
- [x] Как хранить данные: file-first совместимо с SillyTavern, SQLite только как возможный индекс/кеш позже.
- [x] Нужен ли режим мастера/ведущего с особыми правами: нет, пока базовая пошаговая таверна для двоих.
- [x] Должен ли интерфейс поддерживать мобильные устройства с самого начала: да, обязательно.
- [x] Нужна ли совместимость с форматами SillyTavern для персонажей/чатов/пресетов: обязательно.

## Принципы разработки

- [ ] Сначала минимальный рабочий прототип, затем расширение.
- [ ] Конфигурация и данные пользователя не должны быть жестко зашиты в код.
- [ ] Backend и frontend должны быть разделены так, чтобы frontend можно было заменить или развивать отдельно.
- [ ] Локальный запуск должен быть простым и документированным.
- [ ] Сетевые настройки должны быть явными, чтобы пользователь понимал, кто может подключиться.
